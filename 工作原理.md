# Local Code RAG 完整工作原理

整套 Local Code RAG 完整工作原理（结合 local_rag_server.py 代码）

整体分为五大阶段：**初始化建库 → 用户提问 → 向量语义检索 → 拼接约束 Prompt → Ollama 模型生成回答**，全程纯本地运行，数据不出本机。

---

## 一、阶段 1：程序启动 & 向量库自动初始化

### 启动 `python local_rag_server.py`

1. 先执行判断函数 `has_valid_chroma_db`：检测 `chroma_code_db/chroma.sqlite3` 是否存在
   - **无数据库文件** → 触发全量扫描构建索引
   - **已有完整库** → 跳过扫描，直接启动 API 服务

### 扫描逻辑 `scan_project_and_index()`

1. 递归遍历你配置的 `PROJECT_PATH` 项目目录，自动跳过 `build`、`.git` 等无用文件夹
2. 只读取配置后缀文件：`kt/go/java/xml/properties/gradle/md`
3. 单个大文件按 `CHUNK_SIZE=700` 字符切分成小段（**分片 Chunk**），防止单段文本过长、向量失真
4. 每一段代码文本绑定元数据 `meta:{"file":完整文件路径}`，后面检索可以溯源是哪个文件
5. 分批次传给 `nomic-embed-text` 嵌入模型，把文本文字转换成**高维数字向量**
6. 向量、原文、文件路径一起存入 **Chroma SQLite 向量数据库**持久化保存

> **一次构建完成后，后续重启程序不会重复扫描文件，启动速度极快。**

### 核心名词解释

- **嵌入 Embedding**：文字→数字向量，语义相近的代码片段，向量数值距离更近，机器能看懂 "含义相似"
- **Chunk 分片**：代码不能整文件塞向量库，太长会稀释语义，分段匹配精度更高

## 二、阶段 2：用户发起提问（curl/Continue 插件）

以这条提问举例：
> 针对 ChooseDeviceTypeActivity.kt 里的代码，给出可落地的优化改进点

1. 请求到达 `/v1/chat/completions` 接口
2. 代码提取用户最新一句提问文本，送入检索函数 `get_related_code_context()`

---

## 三、阶段 3：语义向量检索（RAG 核心）

1. 用户问题文本同样交给 `nomic-embed-text` 生成**问题向量**
2. Chroma 库内部计算：**问题向量** 和 **库里所有代码分片向量** 的余弦相似度
3. 配置 `TOP_K=1`，只取出相似度最高的 **1 条代码分片**
4. 同步取出这条分片绑定的**原始文件路径**、**原始代码文本**
5. 组装上下文字符串：带上【文件路径】标记 + 分片源码

### 关键点：不是简单关键词匹配文件名，是理解语义匹配

比如你问 "页面跳转逻辑"，哪怕文件名不带 `jump`，只要代码里有跳转相关逻辑，向量距离近就能命中。

---

## 四、阶段 4：构建高强度约束 System 提示词（解决模型乱答问题）

1. 把命中文件完整路径**置顶写死**在系统指令最高优先级位置
2. **硬性规则锁死**：
   - 只能使用本次检索到的本地源码回答
   - 禁止调用模型自身预训练知识库、通用教程、外部示例代码
   - 回答必须带上文件路径，不许臆断文件不存在、回溯多轮历史乱审计

### 拼接结构

```
强制规则 + 锁定文件列表 + 检索到的源码分片 + 用户原始对话消息
```

3. 设定 `temperature=0`：完全关闭模型自由创意发散，只忠实依据给定内容输出

---

## 五、阶段 5：模型生成 + 双层纠错校验（代码独有的加固逻辑）

1. 整套消息发给 `deepseek-coder` 本地模型推理，产出**第一轮回答**
2. **自动校验**：回答文本里有没有包含命中的文件路径
   - ✅ **存在路径**：直接作为最终回复返回
   - ❌ **没有路径**（模型跑偏输出通用内容）：触发**二次强制重生成**
3. **二次纠正 Prompt 强制施压**：只准分析当前 kt 文件源码，禁止外部内容
4. 二次推理结果作为最终应答，包装成标准 OpenAI JSON 格式返回给 curl/VSCode 插件

---

## 六、附加运维接口：`/ingest/rebuild`

代码大批量修改、新增 / 删除文件后调用这个接口：

1. 清空向量库内全部旧数据
2. 重新完整扫描项目、分片、嵌入、入库
3. 不用重启服务，实时刷新代码向量索引

---

## 七、整套链路数据流闭环

```plaintext
项目源码文件 → 分片Chunk → nomic-embed-text生成向量 → Chroma持久存储
       ↓
用户提问 → 提问向量 → 相似度检索 → 匹配代码片段+文件路径
       ↓
片段+强约束Prompt → deepseek-coder 推理 → 路径校验纠错 → 返回回答
```

---

## 对比两个模式差异

| 模式 | 特点 | 优缺点 |
|------|------|--------|
| **纯 RAG（当前这套）** | 语义向量匹配，理解代码逻辑 | 精度高；需要一次初始化建库 |
| **简易文件名匹配（备选方案）** | 只匹配文件名文字 | 不懂代码含义，无向量、无嵌入 |

---

## 为什么 6.7b 模型容易跑偏，代码要加双层校验？

- `deepseek-coder 6.7b` 量化版指令跟随能力弱，即便 prompt 写死约束，依然容易优先调用自身训练里的通用编程知识
- `13b` 及更大体量模型对 system 指令服从度大幅提升，很多场景不需要二次纠错也能严格只看 RAG 源码作答

---

## 安全特性

**全程 100% 本地离线**：代码、向量、问答、大模型推理全部跑在本机 Ollama，没有任何网络上传、云端调用。
