RAG
RAG 是 Retrieval-Augmented Generation 的缩写，中文意思是检索增强生成。

简单来说，它是一种结合了信息检索和大语言模型生成能力的技术，用来解决大模型“不懂未知信息”或“胡编乱造”（幻觉）的问题。

通俗理解（开卷考试 vs 闭卷考试）
普通大模型：像闭卷考试。只能靠训练时记住的知识回答。如果问它没学过的最新信息，它要么答错，要么瞎编。

RAG模型：像开卷考试。接到问题后，它先跑去查资料（检索），找到相关文档，再参考这些资料组织答案（生成）。

工作流程（三个步骤）
检索：系统收到你的问题（如“我的订单什么时候到？”），立刻在知识库（如公司手册、产品文档）中搜索最相关的信息。

增强：把搜到的相关信息，连同你的原始问题，一起打包成一个“提示”，提交给大模型。

生成：大模型基于提供的“参考资料”来生成回答，而不是凭空想象。

为什么需要 RAG？（主要优势）
知识更新快：大模型训练一次成本高，知识往往是旧的。RAG可以实时接入最新文档、数据库或新闻。

减少胡说八道：答案有检索结果作为依据，大大降低了凭空捏造的概率。

答案可验证：模型可以引用信息来源（如“根据第X页文档…”），方便你核实。

处理小众或私有知识：企业可以把内部机密文档（如人事制度、技术规范）做成RAG系统，供内部员工询问，而无需把数据交给大模型训练。

搭建RAG两种方案

💥方案 1：裸写调度（当前优先推荐，零折腾起步）
技术栈：Python + chromadb 原生 API + ollama SDK + FastAPI
封装层级：无中间框架，所有流程自己代码控制
依赖清单：chromadb, ollama, fastapi, uvicorn 四个，极少无冲突
代码量：300 行左右，逻辑直白，扫描→分块→入库→检索→转发对话接口
向量底层：Chroma 内部自带 FAISS 索引，你完全不用操作向量数组
适配场景：单一代码工程知识库、对接 VSCode Continue、只需要基础 RAG 问答
优点：启动快、内存占用低、报错直观、修改参数随手改、M Mac 兼容性拉满
短板：多数据源、工具调用、对话记忆、多路召回等高级能力需要自己逐行手写扩展
💥方案 2：LangChain 标准框架方案
技术栈：Python + LangChain + Chroma + OllamaEmbeddings + ChatOllama + FastAPI
封装层级：全套链式组件，文档加载器、文本分割器、检索链、提示模板全部封装好
依赖清单：langchain、langchain-ollama、langchain-chroma、chromadb、fastapi、uvicorn，包更多，偶尔出现 API 弃用、版本兼容小坑
代码量：比裸写略少一点模板代码，但框架概念多（Chain、Retriever、Template、Memory）要理解
向量底层：同样复用 Chroma 存储，向量库文件和裸写互通，嵌入模型依旧 nomic-embed-text
适配场景：未来要扩展 AI 工具调用（执行 shell、gradle、git）、多文档混合（代码 + md+pdf）、多轮对话记忆、多项目知识库切换
优点：工业标准组件，网上案例极多，高级功能开箱即用，不用重复造轮子
短板：多一层抽象，小问题排错堆栈更深，轻度场景有性能冗余

二、落地顺序（最优稳妥路线，不用二选一纠结）
第一步：先用裸写脚本跑通完整闭环
先把「扫描项目→向量入库→11435 接口给 Continue→提问自动读代码」整条链路调通验证效果。
好处：当天就能用起来写代码，不耽误开发，踩最少环境坑。
第二步：等有扩展需求再无缝迁移 LangChain
向量库chroma_db文件夹完全通用，嵌入模型不变，不用重新索引工程；
只替换后端服务代码，端口依旧 11435，VSCode Continue 配置一字不用改，切换后端程序即可。
